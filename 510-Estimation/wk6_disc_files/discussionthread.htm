The different requirements are going to be grouped into the following categories. Each grouping will have an overall flexibility assigned for it, based on the requirements that it represents.
- Functional Requirements – describe the functionality of the application
- Platform Requirements – the underlying software platform requirements
- Performance and Availability Requirements – how well the application performs
- Security Requirements – security constraints including what’s needed for the application and existing standards


The functional areas for the application will be separated into 4 different categories that are prioritized as follows:

1. Patient Records (inviolate)– this includes patient records, treatment histories, contact information, drug prescriptions, conforming to HL-7 standards and other patient related information.
2. Billing (inviolate)– this functional area includes the integration with the CPS system, access to billing records, billing information, conforming to HIPAA standards, claims and reimbursements
3. Clinician Profiles (relaxed)– this functional area consists of descriptions of the clinician profiles, what services they have been trained to perform and patient referrals
4. Healthcare Knowledge Base (deferred) – used to describe the different procedures that are performed, research materials to assist in diagnosis and other medical resources.


Security Requirements – Given that this is an application that is not only going to be handling billing and contact information, but also patients medical records, the security is of the utmost importance. This being so, there is no flexibility allowed for the requirements contained within this section.
- Public Key Infrastructure for messaging (inviolate)– encrypted communication amongst the available web services
- Secure Socket Layer (inviolate)– encrypted communication between the application and the clients (workstations, handhelds)


Platform Requirements – The updated case study describes that the following should be “sufficiently” covered. I’m taking this to mean that these requirements can be relaxed. Other requirements that were described with a “must” are characterized as inviolate.
- Reliability (relaxed)
- Scalable over 5 years (relaxed or deferred) – the initial design may have to account for it, but depending on how much refactoring we can do between deployments…
- Availability (relaxed)
- Maintainability (relaxed)
- MS Windows support (inviolate)
- Operating System (inviolate)
- Database Management System (inviolate) – including backup and restore functionality


Performance Requirements –
- Response times under 5 seconds (deferred – but may have to be included in the application design and hardware procurement phases)
- Performance monitoring (relaxed)


Functional Requirements – The following are arranged in order from most important to least.
- Patient medical records (inviolate)
- Office workstations (inviolate)
- Integration with CPS (inviolate)
- Conforming to HIPAA and HL-7 standards (inviolate)
- Data conversion (bootstrapping the new application) (inviolate)
- Access control (inviolate)
- System administration (inviolate)
- Patient Registration (relaxed)
- Patient scheduling (relaxed)
- Clinician profiles (relaxed)
- Medical information system (deferred)
- Handheld/PDA support (deferred)
- Appointment scheduling (deferred)
- Online Forms (deferred)
- Personal storage (deferred) – Portable Identity Management
- IM, texting support (deferred)

I listed all requirements areas (with the exception of performance) and identified the probability of:

. COTS
. Reuse
. New tools (increase productivity)
. Benefit from new personnel / training

The factors were rated as high, medium or low. These subjective measures are detailed in the <b>attached spreadsheet</b>. I will summarize the "High" areas here.

All web services (UW, HS, CS) will benefit highly from strengthening tool support since design tools are increasingly supporting this technology. Database design and management will also benefit from tool support; this is a very mature technology area.

All system administration, user identities, access control and management should use COTS. Most operating systems have this capability built in and there are many third party tools (which use open standards such as certificates and Kerberos) to provide access control. The selected technologies should be compatible with the legacy Cascade systems.

Database backup, performance monitoring, logging and general administration is best performed with COTS software. There are many good backup and recovery programs which could do a better job than us for less cost than design, implementation and testing of this capability.

Appointment management is another high candidate for COTS software. Calendaring software and utilities are readily available and they also interface nicely with custom software (such as we would create.)

Portable identity management should be included (or compatible) with the COTS solution we have identified for our web services based system.

Documentation should make good reuse of existing plans and documents from other similar projects. Additionally we could make use of more modern document management technologies (like version control, collaboration technologies, etc.)
