Project II – Annotated Programming Standard

Objective:

To understand how to use a programming standard to both develop software and evaluate code.

Description:

The programming standard, sometimes known as a coding standard, is an invaluable tool for communicating development best practices within a development team. When coupled with explicit evaluation criteria for inspection or review of code it provides a reference for expected style and implementation guidelines that help ensure effective reviews.

Output:

There are two outputs expected from this part of the project:

1.     Annotated Programming standard/style

a.     Answers the question: How should programs be constructed?

b.     Should address naming, layout, commenting, etc.

c.     Include standards for debugging and instrumentation of code.

d.     Control structure use and function construction.

e.     Module construction and File organization should also be addressed.

 

Annotations should be used to clarify the reason for each rule or guideline. In other words, why is this a rule or a guideline? Is it really important for maintainable/reliable/etc. code, or is it simply being included as a matter of preference? Understanding why something is being included will help you build a more effective (and meaningful) programming standard.

 

Notice that there is no language specified for the programming standard. Keep the focus on high-level concepts that are generally (or broadly) applicable irrespective of the implementation language choice.

 

2.      Checklist for Code Inspection/Review

a.     Answers the question: How should construction be evaluated?

b.     Include things such as traceability to design/requirements

c.     Readability, understandability, etc.

d.     Address module level as well as detailed code level.
